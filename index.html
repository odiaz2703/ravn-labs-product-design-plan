Ravn Labs — Internal Proposal

Product mindset framework for dev + design teams

A practical, phased plan for embedding product ownership, discovery skills, and AI-augmented workflows into every Labs sprint — whether your starting point is a Figma file or a PRD in AgentOS.

Introduction

Ravn Labs was created to push engineers and designers beyond execution — toward product ownership, critical thinking, and the ability to grow and adapt as AI-assisted development reshapes what clients expect from a technology partner.

Each Labs sprint is a sandbox to test new tools, challenge assumptions, and build the habit of understanding a problem before solving it. The first hackathon cycle made clear that this mindset — asking the right questions, validating before building, and delivering work others can immediately act on — is a skill that needs to be developed deliberately, not assumed.

This plan gives every Labs team a shared foundation: a way of working that applies from the first read of a PRD to the final demo, and compounds in value with every cycle.

Core principles

Design that speaks to everyone

Designers gather requirements first, then produce workflows, mocks, and specs that are immediately legible — to developers, stakeholders, and clients alike. No guesswork required.

Owners are the dev's client

Developers must ask discovery questions, validate assumptions, and own problem understanding — not just tickets. The stakeholder's problem is the team's responsibility to solve.

AI is the accelerant

AgentOS, BMAD, and Claude Code compress execution — but judgment and product thinking stay human.

5-phase plan
01
Read the PRD like a product owner
Before writing a single line of code or design
  • Read the full PRD and list every assumption the team is making — even obvious ones. Flag anything that feels ambiguous or contradictory.
  • Write 3–5 discovery questions to bring to the product owner or stakeholder. Make them direct and specific, not vague.
  • Use AgentOS or BMAD to ingest the PRD and generate an initial task breakdown. Treat this output as a draft, not a contract — apply judgment on priority and scope.
  • If the PRD doesn't make sense or a better solution exists, document why and open the conversation with the product owner before pivoting. Don't change direction silently.

02
Run a structured discovery session
Before designing or building anything substantive
  • Schedule a short sync with the product owner or stakeholder. Come with the assumption list from phase 01 — not a blank slate.
  • Ask direct, closed questions first to validate assumptions: “Is it correct that users self-select their tools from a company list?” Then use open questions to go deeper.
  • Identify the primary user, their real pain point, and what “done” looks like. Write this in one sentence and confirm it with the owner before the call ends.
  • Record or log the session and use Claude to extract key decisions, open questions, and confirmed next steps as structured notes.

03
Design with stakeholders in mind
Requirements to legible workflows, mocks, and handoffs
  • Translate confirmed requirements into workflows and mockups that any stakeholder can read without explanation — product owners, developers, and clients all need to understand what is being built and why.
  • The developer is the designer's most immediate stakeholder. Every design decision should make the developer's job clearer: what to build, what is reusable, and what the expected outcome looks like must be unambiguous.
  • Organize Figma files by phase or feature. Templates should use swappable component slots — show layout and structure, not finished screens. New phases get new files that import shared components.
  • Create a dedicated dev handoff section with explicit annotations: what is new, what is reused, and current build status. Use a simple checkbox system so developers always know where things stand.
  • For rapid prototyping, use Claude Code or Stitch with ShadCN defaults. The priority is UX and data flow — visual polish comes later. Share working prototypes with developers early and iterate together.
  • Check in with developers regularly. Ask directly: are these designs clear? Is anything ambiguous? Collect that feedback and feed it back into the next design iteration.

04
Build with AI — and enforce shared components
Claude Code, AgentOS, and component discipline
  • Use AgentOS task files or BMAD to structure the sprint. Treat AI-generated task lists as a starting point — apply judgment on priority, dependencies, and scope against the agreed PRD.
  • When Claude Code generates UI, explicitly instruct it on shared components: “We already have a card component — reuse it, don't create a new one.” AI will always generate new components unless told otherwise.
  • Devs and designers run regular component audits: are UI elements consistent? Are new components being introduced without reason? This is a shared responsibility, not just a dev concern.
  • Every feature branch maps to a specific section in the Figma handoff file. Developers mark what has been implemented; designers update component status accordingly.

05
Demo, retro, and carry learnings forward
End of sprint — the red/green light moment
  • The demo is a presentation of a working product — not a status update. Teams should aim to have a functional, demonstrable product by the end of the sprint. This is exactly why understanding the PRD, validating assumptions early, and managing scope from day one are not optional: they are what make a complete demo possible.
  • If the product is not fully complete, the demo should still show real, working functionality — not slides or descriptions of what was intended to be built. Partial working features communicate far more than a walkthrough of what didn't get done.
  • Come to the retro with 3 specific things learned — not vague reflections. Share what AI tools were used, what worked, and what didn't. These insights are what the Labs program compounds over time.
  • If the project is red-lit, leave it in a clear state: update the roadmap file, document where the team ended, and note the recommended next step. There is a real chance this project gets picked up again.
  • If green-lit, the team writes a round-2 scope document — revisiting the PRD with what is now known — before continuing development. Don't carry unresolved assumptions into the next cycle.
  • The project lead collects dev and designer feedback on handoff clarity and shares it as a team improvement note. Every cycle should make the next one sharper.
